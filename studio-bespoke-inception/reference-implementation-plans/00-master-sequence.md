# Studio Bespoke — Reference-Informed Homepage Master Sequence

**Status:** Implementation-ready planning index  
**Selected concept:** The Ritual Plan  
**Physical law:** A settled residence moves slowly, with weight.  
**Scope:** Homepage planning only. This directory does not authorize code modification, asset publication or deployment.

## Outcome

The six references confirm the approved story architecture but expose three improvements needed in the current prototype:

1. The opening, problem and proof must behave as one continuous system rather than three components that only appear related.
2. The homepage needs complete range, process and inquiry chapters; the current build stops after a partial details section.
3. Motion ownership must be reduced and explicit so the proof pin remains the climax instead of one effect among many.

## Behaviours that must remain unchanged

1. Keep the approved opening line exactly: **“A home becomes personal when its plan begins to recognize the life inside it.”**
2. Keep the blueprint as the opening visual plane.
3. The Inherited Plan must not become a solid panel sliding over the opening.
4. No Al Barari image may be used as Mira proof.
5. The proof aperture keeps one arch geometry and scales uniformly; it does not morph.
6. The proof uses one coherent Mira room, not unrelated room cycling.
7. Desktop may pin the proof; mobile and reduced motion use ordinary stacked scenes.
8. Lenis remains the shared desktop scroll infrastructure unless verification proves it harms control or accessibility.
9. Melodrama and Satoshi remain the approved type direction pending font licensing/hosting confirmation.
10. The current Travertine & Ironwood tokens remain the working palette until licensed project imagery is sampled and contrast-tested.

## Deliberate changes approved by this plan

1. Treat “preloader” as an arrival scene, not a reason to delay access.
2. Use one shared plan asset across Opening and Inherited Plan instead of separate drawings.
3. Make the controlling line the semantic `h1`.
4. Replace the partial details grid with a Mira-only decision sequence.
5. Add supporting work, process/trust and qualified inquiry chapters.
6. Repair navigation anchors and replace `Journal` with `Press` unless real journal content is supplied.
7. Remove or block all unverified contact addresses and typo-bearing logo assets from production.
8. Keep critical information visible without hover, animation or JavaScript.

## Implementation order

1. **Foundation:** content data, verified anchors, semantic page outline, header/mobile menu, motion preferences and image conventions.
2. **Continuous first act:** Opening + Inherited Plan as one shared-plan system.
3. **Flagship proof:** refine the existing Mira aperture and evidence timeline.
4. **Mira consequences:** build the four-decision Rituals composition with static assets first, then sticky enhancement.
5. **Range:** add Zulal and Desert Leaf as two art-directed proof chapters.
6. **Trust:** add three delivery stages, founder authority, testimony and capacity.
7. **Inquiry:** add fit-led form, error/success states and restrained footer.
8. **Motion pass:** enable only the approved arrival, handoff, proof pin and sticky detail behaviour.
9. **Whole-page verification:** desktop, mobile, reduced motion, keyboard, slow network, failed media and reverse scroll.

Do not build all motion first. Each chapter must pass as static semantic content before its enhancement is enabled.

## Complete-page acceptance criteria

### Argument

- A first-time visitor can state that Studio Bespoke changes how a home functions, not only how it looks.
- Mira appears as one coherent project with constraint, intervention and consequences.
- Zulal and Desert Leaf extend the method without becoming equal cards in a portfolio dump.
- Process and founder content arrive only after spatial proof.
- Inquiry asks about a real domestic friction and states fit calmly.

### Motion

- Only the opening handoff and Mira proof require GSAP scroll timelines.
- The Mira pin reverses cleanly at slow and fast scroll speeds.
- No scroll state hides navigation, proof or inquiry.
- Mobile contains no pin, horizontal scroller or hover dependency.
- Reduced motion presents the complete argument in ordinary flow.

### Content and evidence

- Every project claim traces to the evidence dossier.
- No fabricated before image, plan or demolition state appears.
- Every Rituals image belongs to Mira.
- Contact details and logo files are confirmed before activation.
- Press attribution is exact and subordinate to project proof.

### Accessibility

- One `h1`, ordered heading levels and semantic sections/landmarks.
- Skip link and visible keyboard focus.
- Menu, annotations, gallery state and form are keyboard and touch operable.
- All meaningful images have accurate alt text; decorative linework is hidden.
- Text remains legible over every image state.

### Performance and resilience

- First viewport loads one media system only.
- Below-fold images are responsive and lazy-loaded.
- No hidden autoplay media continues offscreen.
- Failed video or JavaScript leaves readable static content.
- No layout shift from missing intrinsic media dimensions.
- Development build, lint, desktop/mobile browser checks and reduced-motion checks pass before handoff.

## Pre-mortem

| Failure six months later | Early warning | Prevention in this plan |
| --- | --- | --- |
| Site looks like a collage of reference effects | More than one signature mechanic per chapter; repeated pins | One climax pin, one sticky gallery, normal flow elsewhere |
| Ritual language becomes empty poetry | Captions cannot name a plan/material decision | Every ritual requires a factual decision and matching image |
| Motion works only on desktop | Mobile is designed after desktop timelines | Static/mobile state is the base implementation for every section |
| Page becomes too heavy | Multiple videos, full-resolution remote images, early preload | One opening media system, still-led proof, lazy-loaded later assets |
| Trust collapses at inquiry | Typo-bearing logo or wrong email remains | Contact/logo confirmation gates submission and production |
| Supporting work flattens the position | Projects appear as equal cards without constraints | Two asymmetric decision narratives only |

## Final structural trace

`Opening belief` → `inherited constraint` → `Mira intervention` → `daily consequences` → `method at two other scales` → `delivery responsibility` → `qualified invitation`.

Every implementation plan must preserve that order unless new evidence changes the underlying argument.
